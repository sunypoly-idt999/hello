# AIX Agentic Zettelkasten — Six-Agent Architecture

## Premise

The zettelkasten is an agentic knowledge system. Six agents own six domains. Together they transform raw input — conversations, browsing, listening — into routable, citable, reusable knowledge objects.

---

## The Six Agents

### Orchestrator

Owns direction. Manages the roadmap, the daily log, and session continuity across two parallel tracks. Track A is pipeline development — what gets built, in what order, what is blocked. Track B is course development — what content is needed, for which course, by when. Every session begins and ends with the Orchestrator. It takes orders from Steve and gives orders to all five agents below it.

---

### Context

Owns contexts. Projects, courses, contributors, topics, and the relationships between them. When any other agent asks where something belongs, Context answers. It is the live, queryable version of the registry file. Active projects include herkimer, tdac-ac, pnt, ai-research-workshop, and games-nigeria. Active courses are COM 216, IDT 555, STS 188, and FYS 111. Active contributors are Steve, Asela, Alvin, Bill, Nick, and Trusting.

---

### Source

Owns sources. Every external anchor — DOI, ISBN, podcast URI, YouTube ID, web URL — lives here. Zotero is the system of record. Three equivalent input surfaces: the Zotero API, the Zotero desktop application, and the Zotero browser connector. Deduplication uses DOI as the primary key, ISBN as secondary, and a URL-derived canonical URI as fallback. Domain-specific extraction rules apply per source type: YouTube video ID, podcast episode URI from RSS feed, DOI via doi.org, ISBN via isbnlib, arXiv paper ID, GitHub repo path. Primary export format is JSON. BibTeX is available for downstream compatibility. The unified bibliography lives in its own repository, separate from all web-facing repos.

---

### Artifact

Owns generated objects. Zets, podshares, and webzets are all artifacts. The Artifact agent runs the capture skills — slash-zet, slash-webzet, and slash-podshare. It collects input via structured form: title, body, projects, type, tags, related URIs. It generates YAML frontmatter and tracks provenance on every output: capture mode, which is live or post-hoc; model name; and session identifier. Key distinction: Source owns what Steve finds in the world. Artifact owns what Steve makes from his own agency.

---

### Relevance

Owns contextualizations. It is the automated curation gate. Every artifact generated by the Artifact agent passes through Relevance before entering the knowledge infrastructure. Three questions per artifact: Is this substantive, or is it noise? Is this unique, or does it duplicate something already in the vault? Is this routable — does it belong to a named project or course? Three possible decisions: publish, which routes to the Publication agent; hold, which flags for Steve's review; or discard, which drops low-signal captures. The Relevance agent also detects emergent concepts worth standalone capture and flags them — for example, a dynamic pair in a podshare record that has zettel value on its own.

---

### Publication

Owns publications. Takes validated artifacts from Relevance and routing targets from Context, then routes to destinations. Output formats include markdown for the vault, HTML for course sites, BibTeX for academic tools, YAML for the knowledge graph, and slide fragments for presentations. Destinations include the zettel repo for atomic notes and conversation summaries, course subsites for teaching content, the bibliography repo for scholarly records, and the Obsidian vault for knowledge graph nodes.

---

## Pipeline

Steve generates via conversation, browsing, or listening. The Artifact agent captures and structures the object. The Source agent normalizes URIs and enriches Zotero records. The Relevance agent validates, scores, and decides. The Publication agent routes to the right destination in the right format. The Context agent sits above the entire pipeline, consulted by all agents for routing decisions.

---

## Skill Extensions Required

slash-podshare currently produces chat output only. It needs to write to Zotero via the Source agent and route via Publication.

slash-webzet currently produces markdown with a Wayback Machine archive URL. It needs to write to Zotero and pass through Relevance.

slash-zet currently produces markdown. It needs form-based structured input and slide fragment output as an additional format.

slash-hello currently loads the registry. It needs to boot the Orchestrator and surface the daily log.

slash-goodbye currently zips outputs. It needs to trigger Relevance scoring on all artifacts captured in the session.

## Build Roadmap

Immediate work: publish this spec, create the aix-bibliography repo and migrate unified.bib, and extend all skill outputs with provenance fields — capture mode, model name, session identifier.

Near-term work: build Zotero API integration for read and write with DOI and ISBN deduplication, build the URI normalization module with domain-specific extraction rules, build a Relevance agent proof-of-concept to score existing podshares, and build the Orchestrator daily log in standup format.

Deferred work: full Publication agent with automated multi-format output and repo commits, yt-dlp integration in Cowork for local YouTube and podcast URI extraction, and batch AIXZ processing of the 761-conversation backlog.

## Daily Log Format

Date. Track A, pipeline: what was built or decided, blockers, next action. Track B, courses: what content was generated or needed, blockers, next action.
